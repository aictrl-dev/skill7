<objective>
Fix the fire-and-forget `.catch(() => {})` bug on `SessionPrompt.prompt()` and `SessionPrompt.command()` in the local SDK stub within `run.ts`, so that errors propagate and cause `aictrl run` to exit non-zero instead of hanging silently.

Purpose: This is the #1 bug that caused real silent CI failures. When `SessionPrompt.prompt()` throws (e.g., session not found, model resolution failure, network error), the `.catch(() => {})` silently eats the error and the event loop hangs forever waiting for a `session.status` idle event that will never arrive.

Output: Modified `run.ts` with error-propagating prompt/command calls, plus a regression test.
</objective>

**Step 1: Add a promise variable before the local SDK stub definition.**

Inside `execute()`, before the local SDK object is defined (before line 689), declare:

let promptResult: Promise<any> = Promise.resolve()

**Step 2: Fix the `prompt()` method in the local SDK stub (lines 713-720).**

Replace:
```typescript
async prompt(opts: any) {
  SessionPrompt.prompt({
    sessionID: opts.sessionID,
    parts: opts.parts,
    agent: opts.agent,
    model: opts.model,
    variant: opts.variant,
  }).catch(() => {})
},
```

With:
```typescript
async prompt(opts: any) {
  promptResult = SessionPrompt.prompt({
    sessionID: opts.sessionID,
    parts: opts.parts,
    agent: opts.agent,
    model: opts.model,
    variant: opts.variant,
  })
},
```

No `.catch(() => {})`. Store the promise in `promptResult`. Do NOT await it (fire-and-forget semantics must be preserved for the event loop pattern).

**Step 3: Fix the `command()` method in the local SDK stub (lines 722-730).**

Apply the same fix -- store the promise in `promptResult`, remove `.catch(() => {})`:
```typescript
async command(opts: any) {
  promptResult = SessionPrompt.command({
    sessionID: opts.sessionID,
    command: opts.command,
    arguments: opts.arguments ?? "",
    agent: opts.agent,
    model: opts.model,
    variant: opts.variant,
  })
},
```

**Step 4: Replace `await loopDone` with a race against the prompt promise (around line 680).**

Replace:
```typescript
await loopDone
```

With:
```typescript
// Race loopDone against promptResult to handle early prompt failures.
// If SessionPrompt.prompt() rejects BEFORE it enters its internal loop()
// (e.g., Session.get() fails, model not found), no session.status idle event
// is emitted, so loopDone would hang forever. Racing ensures we surface
// the error and exit.
await Promise.race([
  loopDone,
  promptResult.then(
    // If prompt resolves normally, wait for the event loop to finish
    () => loopDone,
    // If prompt rejects, surface the error immediately
    (e) => {
      error = error ? error + EOL + String(e) : String(e)
      emit("session_complete", {
        durationMs: Date.now() - startTime,
        error: String(e),
      })
      console.error(e)
      process.exit(1)
    },
  ),
])
```

**CRITICAL: The `EOL` import already exists at the top of run.ts (imported from "os").** Verify this before making changes.

**What to avoid and WHY:**
- Do NOT `await` the prompt/command call directly in the SDK stub. The event loop (`loop()`) must start first and consume bus events concurrently. Awaiting would deadlock the two.
- Do NOT use `Promise.all([loopDone, promptResult])` -- if promptResult rejects, `loopDone` still hangs forever (the idle event never arrives). `Promise.all` waits for ALL promises, so it would hang. The `Promise.race` pattern with the `.then()` transformation is correct because: on success, we fall through to waiting for `loopDone` normally; on failure, we exit immediately.
- Do NOT add a generic `.catch()` on `promptResult` outside the race -- that would prevent the rejection from being visible in the race.

Create a regression test that verifies `aictrl run` exits non-zero when the prompt fails.

The test should use the subprocess pattern (spawn a process, check its exit code) since we need to verify `process.exit` behavior. Follow the pattern from `test/util/process.test.ts`.

```typescript
import { describe, expect, test } from "bun:test"
import path from "path"
import { fileURLToPath } from "url"

const __dirname = path.dirname(fileURLToPath(import.meta.url))
const CLI_ENTRY = path.resolve(__dirname, "../../src/index.ts")

describe("aictrl run error propagation", () => {
  test("exits non-zero when prompt fails with invalid model", async () => {
    // Run aictrl with a model that does not exist, causing SessionPrompt.prompt
    // to throw during model resolution. Before the fix, this would hang forever
    // because .catch(() => {}) swallowed the error and no idle event was emitted.
    const proc = Bun.spawn(
      [
        "bun",
        "run",
        "--conditions=browser",
        CLI_ENTRY,
        "run",
        "test prompt",
        "--model",
        "nonexistent-provider/nonexistent-model",
        "--print-logs",
      ],
      {
        cwd: process.cwd(),
        env: {
          ...process.env,
          // Ensure we don't accidentally use real API keys
          ANTHROPIC_API_KEY: "",
          OPENAI_API_KEY: "",
        },
        stdout: "pipe",
        stderr: "pipe",
      },
    )

    // Give the process a generous timeout -- if it hangs (the old bug), this
    // test will fail by timeout rather than passing
    const timeout = setTimeout(() => {
      proc.kill()
    }, 15_000)

    const exitCode = await proc.exited
    clearTimeout(timeout)

    // The process must exit non-zero (1) when the prompt fails
    expect(exitCode).not.toBe(0)
  }, 20_000)

  test("does not contain .catch(() => {}) on critical path", async () => {
    // Static analysis: ensure the fire-and-forget pattern is not reintroduced
    const runSource = await Bun.file(
      path.resolve(__dirname, "../../src/cli/cmd/run.ts"),
    ).text()

    // Search for the specific antipattern: .catch(() => {}) on SessionPrompt calls
    const hasSilentCatch = /SessionPrompt\.(prompt|command)\([^)]*\)[^.]*\.catch\(\(\)\s*=>\s*\{\s*\}\)/.test(
      runSource,
    )
    expect(hasSilentCatch).toBe(false)
  })
})
```

**Key design decisions:**
- Uses `Bun.spawn` to run the CLI as a subprocess, matching the project's subprocess testing pattern.
- Uses `--conditions=browser` flag which is how the CLI is run from source (see CI config in MEMORY.md).
- Tests with an invalid model (`nonexistent-provider/nonexistent-model`) which will cause prompt() to throw during model resolution.
- Sets a 15-second kill timeout: if the old bug is present, the process would hang forever; the timeout prevents the test from hanging too.
- The 20-second test timeout (second arg to `test()`) is slightly longer than the kill timeout.
- Includes a static analysis test that greps the source for the `.catch(() => {})` antipattern to prevent regression.
- Does NOT require `Instance.provide()` because we are spawning a separate process.

**What to avoid:**
- Do NOT try to mock `SessionPrompt.prompt` -- this is a process-level exit test, not a unit test.
- Do NOT use `Process.run` from `src/util/process.ts` -- it would throw `RunFailedError` on non-zero exit which complicates the assertion. `Bun.spawn` with `.exited` is cleaner.
