<objective>
Fix the `uncaughtException` and `unhandledRejection` handlers in both entrypoints (`index.ts` and `headless.ts`) to call `process.exit(1)` after logging and to preserve full stack traces.

Purpose: These handlers are the last safety net. Currently they log only `e.message` (losing the stack trace) and do NOT call `process.exit(1)`, so the process continues running in an undefined state after an uncaught exception. This causes `aictrl run` CI jobs to appear hung instead of failing cleanly.

Output: Modified `index.ts` and `headless.ts` with fixed handlers, plus a regression test.
</objective>

**Changes per handler (2 lines added each):**
1. Add `stack: e instanceof Error ? e.stack : undefined` to the log call -- preserves the full stack trace for debugging
2. Add `process.exit(1)` after the log call -- ensures the process terminates in a defined state

**What to avoid and WHY:**
- Do NOT use `process.exitCode = 1` instead of `process.exit(1)`. After an uncaught exception, the process is in an undefined state. Setting `exitCode` allows the event loop to continue, which is exactly the bug we are fixing. `process.exit(1)` is the correct behavior per Node.js/Bun documentation.
- Do NOT add any cleanup logic (like flushing logs) between the log call and `process.exit(1)`. Cleanup after uncaught exception is unreliable and can cause secondary failures. The existing `Log.Default.error()` call is synchronous enough for the log to be written.
- Do NOT change the existing try/catch/finally block at the bottom of index.ts or headless.ts -- that is the NORMAL error path and works correctly. These handlers are the LAST RESORT safety net.

Create a regression test that verifies the exception handlers exit the process with code 1.

Since we cannot trigger a real uncaught exception inside a test runner (it would kill the test process), use the subprocess spawn pattern: create a small inline script that registers the handler and then throws, then verify it exits with code 1.

```typescript
import { describe, expect, test } from "bun:test"

describe("exception handler exit behavior", () => {
  test("uncaughtException handler exits with code 1", async () => {
    // Spawn a minimal script that:
    // 1. Registers the same exception handler pattern as index.ts
    // 2. Throws an uncaught exception
    // 3. Should exit with code 1 (not continue running)
    const script = `
      process.on("uncaughtException", (e) => {
        process.stderr.write(e instanceof Error ? e.stack || e.message : String(e))
        process.exit(1)
      })
      // Throw after a tick to ensure it's truly "uncaught"
      setTimeout(() => {
        throw new Error("test uncaught exception")
      }, 10)
    `
    const proc = Bun.spawn(["bun", "-e", script], {
      stdout: "pipe",
      stderr: "pipe",
    })

    const exitCode = await proc.exited
    const stderr = await new Response(proc.stderr).text()

    expect(exitCode).toBe(1)
    expect(stderr).toContain("test uncaught exception")
    // Verify stack trace is preserved
    expect(stderr).toContain("Error:")
  }, 10_000)

  test("unhandledRejection handler exits with code 1", async () => {
    const script = `
      process.on("unhandledRejection", (e) => {
        process.stderr.write(e instanceof Error ? e.stack || e.message : String(e))
        process.exit(1)
      })
      // Create an unhandled rejection
      Promise.reject(new Error("test unhandled rejection"))
    `
    const proc = Bun.spawn(["bun", "-e", script], {
      stdout: "pipe",
      stderr: "pipe",
    })

    const exitCode = await proc.exited
    const stderr = await new Response(proc.stderr).text()

    expect(exitCode).toBe(1)
    expect(stderr).toContain("test unhandled rejection")
  }, 10_000)

  test("index.ts exception handlers include process.exit(1)", async () => {
    // Static analysis: ensure handlers include process.exit(1)
    const indexSource = await Bun.file(
      require.resolve("../../src/index.ts"),
    ).text()

    // Verify both handlers exist and include process.exit(1)
    // Match the pattern: process.on("uncaughtException", ...) block containing process.exit(1)
    expect(indexSource).toContain('process.on("uncaughtException"')
    expect(indexSource).toContain('process.on("unhandledRejection"')

    // Verify process.exit(1) appears after both handlers (crude but effective)
    const uncaughtIdx = indexSource.indexOf('process.on("uncaughtException"')
    const rejectionIdx = indexSource.indexOf('process.on("unhandledRejection"')

    // Find the next process.exit(1) after each handler registration
    const exitAfterUncaught = indexSource.indexOf("process.exit(1)", uncaughtIdx)
    const exitAfterRejection = indexSource.indexOf("process.exit(1)", rejectionIdx)

    // Both should find process.exit(1) within the handler (before the next major block)
    expect(exitAfterUncaught).toBeGreaterThan(uncaughtIdx)
    expect(exitAfterRejection).toBeGreaterThan(rejectionIdx)
    // Handlers are close together, so exit should be within ~200 chars
    expect(exitAfterUncaught - uncaughtIdx).toBeLessThan(200)
    expect(exitAfterRejection - rejectionIdx).toBeLessThan(200)
  })

  test("headless.ts exception handlers include process.exit(1)", async () => {
    const headlessSource = await Bun.file(
      require.resolve("../../src/headless.ts"),
    ).text()

    expect(headlessSource).toContain('process.on("uncaughtException"')
    expect(headlessSource).toContain('process.on("unhandledRejection"')

    const uncaughtIdx = headlessSource.indexOf('process.on("uncaughtException"')
    const rejectionIdx = headlessSource.indexOf('process.on("unhandledRejection"')

    const exitAfterUncaught = headlessSource.indexOf("process.exit(1)", uncaughtIdx)
    const exitAfterRejection = headlessSource.indexOf("process.exit(1)", rejectionIdx)

    expect(exitAfterUncaught).toBeGreaterThan(uncaughtIdx)
    expect(exitAfterRejection).toBeGreaterThan(rejectionIdx)
    expect(exitAfterUncaught - uncaughtIdx).toBeLessThan(200)
    expect(exitAfterRejection - rejectionIdx).toBeLessThan(200)
  })

  test("exception handlers log stack traces", async () => {
    // Verify the handlers include stack trace logging
    const indexSource = await Bun.file(
      require.resolve("../../src/index.ts"),
    ).text()
    const headlessSource = await Bun.file(
      require.resolve("../../src/headless.ts"),
    ).text()

    // Both should log e.stack
    for (const source of [indexSource, headlessSource]) {
      const handlerSection = source.slice(
        source.indexOf('process.on("unhandledRejection"'),
        source.indexOf("let cli"),
      )
      expect(handlerSection).toContain("e.stack")
    }
  })
})
```

**Key design decisions:**
- The subprocess tests (first two) verify the handler PATTERN works correctly (exits on exception/rejection). They use inline scripts, not the real CLI entrypoint, because we only need to verify the handler behavior in isolation.
- The static analysis tests (last three) verify that the real entrypoint source files contain `process.exit(1)` in the right place and log stack traces. This prevents regressions even if someone changes the handler.
- Uses `require.resolve()` for file paths which works in the test environment.
- Test timeouts are generous (10s) for subprocess tests.

**What to avoid:**
- Do NOT import and execute index.ts or headless.ts inside the test process -- they register global handlers that would interfere with the test runner.
- Do NOT use `process.test.ts`'s `Process.run` utility -- it throws `RunFailedError` on non-zero exit, complicating assertions. `Bun.spawn` is cleaner for exit code checks.
